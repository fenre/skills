# Splunk Ingest Actions Reference

Ingest Actions is a Splunk Platform feature for routing, filtering, and masking
data while it streams to the indexers — pre-index processing expressed as
**rules** grouped into a **ruleset**. Rulesets apply to a source type and run in
order. After a ruleset is applied, the data cannot be reverted to its original
form, so filter (drop) rules are destructive.

## Where Ingest Actions Runs

- Splunk Cloud Platform: configure from the search head. On the Victoria
  Experience the ruleset deploys automatically; on Classic you deploy explicitly.
- Indexer clusters: configure on the cluster manager (or a connected search head
  that proxies to it) and deploy the bundle to peers.
- Standalone indexers / heavy forwarders: effective immediately on save.
- Heavy forwarders via deployment server: deployed through the
  `IngestAction_AutoGenerated` server class. S3 destinations must be configured
  on each forwarder individually.

## Rules

- **Filter** — drop events matching a regex/eval. Destructive. Subsequent rules
  apply only to remaining events.
- **Mask** — replace matched substrings (regex) with a replacement, redacting
  sensitive data in `_raw`.
- **Set index** — change the destination index for matching events.
- **Route to destination** — send matching events to a Splunk index and/or an
  RFS (S3 or filesystem) destination. Supports "clone events and apply more
  rules" to split the stream.

## Managing Rulesets (Important)

Create and modify rulesets only through:

- the Ingest Actions page (**Settings > Ingest Actions**), or
- the REST endpoint `/services/data/ingest/rulesets`.

Do **not** hand-edit rulesets into `transforms.conf`. In props.conf the feature
uses the `RULESET` setting (analogous to `TRANSFORMS`): if both a `TRANSFORMS`
and a `RULESET` stanza apply to the same source type, `TRANSFORMS` runs first,
and a source type may be associated with only one `RULESET`.

This skill renders a `ruleset.json` plan to drive the UI or a REST `POST`, rather
than emitting ruleset conf directly.

## RFS Destinations (outputs.conf)

RFS (Remote File System) destinations *are* documented as directly editable via
`outputs.conf` for advanced configuration. The rendered `[rfs:<name>]` stanza
covers:

- `path` — `s3://bucket/prefix` or a filesystem path.
- `remote.s3.endpoint`, `remote.s3.auth_region`, `remote.s3.encryption`,
  `remote.s3.kms.key_id` — S3 settings (similar to SmartStore S3).
- `format` (`raw`/`json`), `compression`, `partitionBy`,
  `batchSizeThresholdKB`, `dropEventsOnUploadError`.

S3 access keys, when unavoidable, are rendered as file-substituted placeholders;
the apply step substitutes them from local files and `chmod 600` the result.
Prefer IAM roles or instance profiles over static keys.

Limits: up to eight S3 destinations and eight filesystem destinations. If a
ruleset routes to a destination that is invalid or does not exist, Splunk blocks
all queues and pipelines (it does not drop data). Always create the destination
before the routing rule.

Tune S3 throughput with `rfs.provider.max_workers` in `limits.conf`. Consider
`dropEventsOnUploadError = true` only for non-critical outputs.

## Manual Heavy-Forwarder Preview

For the classic pre-index path outside the Ingest Actions UI, the rendered
`props_transforms_preview.conf` shows the equivalent `props.conf`
(`TRANSFORMS-...`) and `transforms.conf` stanzas:

- Filter to `nullQueue`: `DEST_KEY = queue`, `FORMAT = nullQueue`.
- Mask `_raw`: `SOURCE_KEY = _raw`, `REGEX`, `FORMAT`, `DEST_KEY = _raw`.
- Set index: `DEST_KEY = _MetaData:Index`, `FORMAT = <index>`.

This is the manual path (useful on heavy forwarders and indexers) and is clearly
labeled; it is not an Ingest Actions ruleset.

## Status And Troubleshooting

`status.sh` lists rulesets (`/services/data/ingest/rulesets`), surfaces RFS/S3
upload errors from `_internal` (`RfsOutputProcessor`, `S3Client`), and shows
effective `[rfs:...]` destinations via `btool` (with key material filtered out).

For S3 auth, follow the SmartStore-on-S3 security guidance; Ingest Actions
requirements are similar.

## Out Of Scope And Handoffs

- Index creation for set-index/route targets: `splunk-hec-service-setup` (HEC),
  `splunk-cloud-acs-admin-setup` (Cloud indexes), or
  `splunk-index-lifecycle-smartstore-setup`.
- Reusable SPL2 pipeline logic and SPL-to-SPL2 review: `splunk-spl2-pipeline-kit`.
- Edge Processor / Ingest Processor pipeline routing: `splunk-edge-processor-setup`,
  `splunk-ingest-processor-setup`.
- Cold-to-archive lifecycle (DDAA/DDSS): `splunk-ddaa-archive`.
