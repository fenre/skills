# Discovery Workflow

`--discover-workloads` is a read-only helper for the first-time operator who does not yet have a populated spec. It walks the cluster, inventories candidate workloads, and probes for the base collector. Output lands under `discovery/`:

```
splunk-observability-k8s-auto-instrumentation-rendered/
  discovery/
    workloads.yaml            # starter inventory with language column left blank
    base-collector-probe.json # helm release + CRD presence
```

## What it reads

- `kubectl get deploy,sts,ds -A -o json` — every workload in every namespace except the default set of system namespaces (`kube-system`, `kube-public`, `kube-node-lease`, `gmp-system`, `gke-managed-*`).
- `kubectl get crd instrumentations.opentelemetry.io -o json` — CRD presence (the single strongest signal the base collector is installed).
- `helm list -A -o json` — helm release named `splunk-otel-collector` (or whatever `--base-release` sets).

## What it never writes

`--discover-workloads` never runs `kubectl apply`, `kubectl patch`, `kubectl delete`, or `helm *` in a mutating form. The mode is safe to run on read-only credentials.

## Inventory schema

`discovery/workloads.yaml` is a YAML list of objects, one per candidate workload:

```yaml
# Generated by splunk-observability-k8s-auto-instrumentation-setup --discover-workloads
# Edit the `language:` field on each entry before running --render --inventory-file workloads.yaml.
workloads:
  - kind: Deployment
    namespace: payments
    name: payments-api
    containers: [app]
    node_selector: {}
    current_annotations:
      prometheus.io/scrape: "true"
    language: ""         # <-- operator fills this in (java | nodejs | python | dotnet | go | apache-httpd | nginx | sdk | skip)

  - kind: StatefulSet
    namespace: payments
    name: ledger-db
    containers: [postgres]
    current_annotations: {}
    language: skip       # explicit skip

  - kind: Deployment
    namespace: checkout
    name: checkout-web
    containers: [web, sidecar]
    current_annotations: {}
    language: ""
```

## Operator workflow

1. Run `--discover-workloads`.
2. Open `discovery/workloads.yaml`.
3. For each entry, fill in `language:` with one of the supported values, or `skip` for workloads that should not be instrumented (databases, caches, static-site servers, etc.).
4. For any entry with multiple containers, add `container_names:` with a comma-separated list of just the app containers (skip sidecars).
5. For Go entries, add `go_target_exe: /path/to/binary`.
6. Re-run the skill with `--render --inventory-file discovery/workloads.yaml`.

## Base collector probe

`discovery/base-collector-probe.json` records:

```json
{
  "helm_release": {
    "present": true,
    "name": "splunk-otel-collector",
    "namespace": "splunk-otel",
    "chart": "splunk-otel-collector-0.130.0",
    "status": "deployed"
  },
  "crds": {
    "instrumentations.opentelemetry.io": true,
    "opentelemetrycollectors.opentelemetry.io": true
  },
  "operator_pod": {
    "present": true,
    "namespace": "splunk-otel",
    "ready": true
  }
}
```

If `helm_release.present` is `false` or `crds.instrumentations.opentelemetry.io` is `false`, the agent must run `splunk-observability-otel-collector-setup` first. The rendered `handoff-collector.sh` carries the exact command.

## Merging into an existing spec

After populating `workloads.yaml`, you can merge it into a full spec file manually or pass the file directly:

```bash
bash skills/splunk-observability-k8s-auto-instrumentation-setup/scripts/setup.sh \
  --render \
  --inventory-file splunk-observability-k8s-auto-instrumentation-rendered/discovery/workloads.yaml \
  --realm us0 --cluster-name prod --deployment-environment prod \
  --languages java,nodejs,python
```

The `--languages` CLI flag and `--inventory-file` compose: languages restrict what CR blocks get emitted, the inventory file drives which workloads get annotated.
